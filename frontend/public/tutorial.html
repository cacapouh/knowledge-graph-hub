1

現場の課題を整理する

まず、サーバー管理で日常的に発生している「あるある」を整理しましょう。

😩 「このサーバー、SIN? SOUT?」

開発チームがサービスイン / サービスアウトの状態に自信が持てず、インフラチームに毎回確認する。Slack が飛ぶ → 返事を待つ → 作業再開、で 30分〜1時間ロス。

😩 「誰に聞けばいい？」

インフラチーム側も、障害時に開発チーム A なのか B なのか分からず、連絡先探しに10分。深夜のオンコールで致命的。

😩 「ログどこにあるの？」

新メンバーやヘルプで入った人が /var/log/app/ なのか /opt/service/logs/ なのか分からない。AI エージェントも当然分からない。

😩 「SSH のやり方が暗黙知」

踏み台経由 → ProxyJump → ポートフォワード… 手順は先輩の頭の中だけ。ドキュメントは古くて信用できない。

💡 共通の根本原因: サーバー情報（状態・ログ・アクセス方法）と組織構造（どのチームがどのサーバーを管理）が 別々の場所にバラバラに存在 し、横断的に引ける仕組みがない。

8

Team → Server リンクを作る

2つのドメインを接続する リンクタイプ を作ります。これがオントロジーの真骨頂です。

⚠️ 現在 UI にリンクタイプ作成フォームがないため、API で作成します。
ブラウザの DevTools コンソール (F12) で以下を実行: 
// Team (id=2) → Server (id=1) の "owns" リンク
fetch('/api/ontology/link-types', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'owns',
    api_name: 'team_owns_server',
    project_id: 1,
    source_object_type_id: 2,
    target_object_type_id: 1,
    cardinality: 'one_to_many'
  })
}).then(r => r.json()).then(console.log)
💡 source/target の ID 確認: Object Type の ID は作成順に振られます。Server を先に作った場合は id=1、Team が id=2 です。Ontology ページの URL やカード表示で確認してください。
Ontology → Link Types タブに「owns」(Team → Server, one_to_many) が表示される。

Object Explorer で確認する

構築したナレッジグラフを Explorer で検証しましょう。以下のシナリオを試してください:

✅ シナリオ1: 「api-prod-02 は SIN? SOUT?」

Server Explorer → api-prod-02 をクリック → 右パネルに status: SOUT と purpose: メモリ増設のためSOUT中 が表示される。Slack で聞く必要なし。

✅ シナリオ2: 「api-prod-02 の件、誰に連絡すれば？」

api-prod-02 の Links → Team「開発チーム Bravo」→ slack_channel: #dev-bravo, lead: 山田 花子。連絡先が即判明。

✅ シナリオ3: 「web-prod-01 のログどこ？」

Server Explorer → web-prod-01 → log_path: /var/log/nginx/access.log, /opt/app/logs/app.log。新人でも即アクセス可能。

✅ シナリオ4: 「batch-prod-01 に入りたいんだけど…」

Server Explorer → batch-prod-01 → access_method: ssh -J bastion.internal batch-prod-01 (要: batch-admins グループ)。踏み台の手順 + 必要な権限まで記載。

11

MCP × Copilot で使うイメージ

このナレッジグラフを MCP (Model Context Protocol) サーバー として公開すると、GitHub Copilot から以下のようなやりとりが可能になります:

👤 開発者: api-prod-02 って今サービスインしてる？
🤖 Copilot:
MCP → GET /api/ontology/objects?type=server → filter hostname="api-prod-02"
api-prod-02 は現在 SOUT（サービスアウト）です。
理由: メモリ増設のためSOUT中
担当: 開発チーム Bravo (#dev-bravo / リーダー: 山田 花子)
👤 開発者: web-prod-01 でエラーが出てるんだけど、ログを見たい
🤖 Copilot:
MCP → GET /api/ontology/objects?type=server → filter hostname="web-prod-01"
web-prod-01 のログはここにあります:
• Nginx: /var/log/nginx/access.log
• アプリ: /opt/app/logs/app.log

アクセス方法: ssh -J bastion.internal web-prod-01
👤 インフラ (深夜オンコール): batch-prod-01 で cron が失敗してる。どこのチームに連絡すれば？
🤖 Copilot:
MCP → GET /api/ontology/objects/{id}/links → resolve team
batch-prod-01 の管理チーム: インフラチーム
→ これはインフラチーム自身の管轄です。
Slack: #infra-ops / リーダー: 中村 健一
オンコール: PagerDuty infra-oncall エスカレーション

※ cron ログ: /var/log/cron.log
👤 マネージャー: 開発チーム Bravo が管理しているサーバーで SOUT 中のものはある？
🤖 Copilot:
MCP → GET Team "Bravo" → links → filter status="SOUT"
開発チーム Bravo の管理サーバーで SOUT 中:
• api-prod-02 (10.1.1.22) — メモリ増設のためSOUT中

他1台 (api-prod-01) は SIN で正常稼働中です。
💡 ポイント: これは単なる「サーバー一覧の検索」ではありません。Team → Server のリンクを辿って横断クエリしている点がナレッジグラフの価値です。従来のスプレッドシートやCMDBでは「BravoチームのSOUTサーバー」を一発で引くのは困難でした。

次のステップ: グラフを拡張する

▶ Middleware オブジェクトタイプを追加
DB (MySQL/PostgreSQL)、キャッシュ (Redis)、MQ (Kafka) をオブジェクトタイプとして追加し、Server → runs → Middleware のリンクを張る。「このサーバーで動いてる MySQL のバージョンは？」に即答可能に。
▶ Runbook オブジェクトタイプを追加
障害対応手順書をオブジェクト化し、Server → has_runbook → Runbook で接続。Copilot が「このエラーの対処法は？」に対して手順書を返せるようになる。
▶ パイプラインで自動同期
Ansible/Terraform の出力 → パイプライン → Server オブジェクト自動更新。SIN/SOUT の変更も自動反映されるため、グラフが常に最新に。
▶ MCP サーバーを実装する
この Foundry の REST API (/api/ontology/*) を MCP ツールとしてラップする。GitHub Copilot の .vscode/mcp.json に登録すれば、Copilot Chat からナレッジグラフに直接クエリ可能。
